# Анализ кода OptiCIP Dashboard — баги, ошибки, оптимизация

> Дата анализа: 2026-07-15. Метод: параллельный разбор 5 модулей, каждая находка проверена в контексте реального кода. Все 145 тестов (`pytest`) на момент анализа проходят.

## Статус исправлений (2026-07-15)

Исправлено и проверено (**147 тестов зелёные**, сервер поднимается, эндпоинты 200, backend-фиксы проверены точечно).

**Приоритет 1 (все):** `int(float(...))` для кодов; `wash-details` снимает снимок под локом и строит деталь снаружи; сортировка объектов через `String(... || "")`; хелпер `fetchWithTimeout` (30 c, `AbortController`) на всех 15 `fetch`.

**Приоритет 2:** №5 — loading-окно показывается ПЕРВЫМ, старт сервера ушёл в `load_desktop_window_url` после `shown` (заодно закрыт П3-пункт про `server.stop()` мимо `finally`; `stop()` устойчив к непущенному потоку); №6 — перебор всех адресов `getaddrinfo`, `SO_REUSEADDR`, обход `PORT=0`; №8 — `(payload.panels || [])`; №9 — рекурсивный `_deep_merge_settings`. №7 — **разобрано, правка не требуется** (сырой минимум концентрации нигде в UI как метрика качества не выводится).

**Приоритет 3 (все применимые):** `Math.min/max` через цикл (RangeError); удалён мёртвый `renderSeriesBadges`; `showToast` гасит авто-таймер; `runHandler` на fire-and-forget `openWashModal`/`openSettings`; debounce 120 мс на `resize`; `logger.warning` на битый `wash_object_names.json`; `atexit`-очистка fallback-`mkdtemp`; `CloseHandle` для мьютекса; ключ дедупа `merge_channel_samples` → `(ts, object_id, process, program)`; настоящая медиана на чётной выборке; **HMAC-подпись записей pickle-кэша** (defense-in-depth, ключ в `0700`-каталоге); хеш пути в fallback-имени FTP-зеркала; `parse_cycle_key` через `rsplit`; явный маркер изолированной точки графика; живое обновление типа линии без полного `render()`; тултип скрывает значение через большой временной зазор.

Новые регрессионные тесты: отклонение подделки HMAC; deep-merge вложенных настроек.

**Оставлено осознанно (1 пункт):** магический `time.sleep(0.35)` в `run_wash_desktop.py` — событие `shown` уже ожидается, а замена native-таймингов открытия окна не проверяема без Windows/webview-окружения; риск регрессии (мигание/центрирование) выше пользы. Помечено как намеренная эвристика.

## Общая оценка

Кодовая база зрелая и написана оборонительно:
- path-traversal в распаковке архивов и FTP-зеркале закрыт (`safe_archive_member_path` + `resolve()`/`is_relative_to`);
- секреты через DPAPI, кэш в приватном `0700`-каталоге с проверкой владельца;
- глобальное состояние под `state_lock`, атомарная запись файлов, LRU-лимиты кэшей;
- фронт: снятие таймеров на `beforeunload`, все вставки данных через `escapeHtml`/Jinja autoescape.

Грубых уязвимостей и утечек нет. Ниже — реальные дефекты по убыванию практической ценности.

## 🔴 Приоритет 1 — чинить (подтверждено в коде)

### 1. `wash_report.py:612-614` — строгий `int()` для кодов роняет строку целиком → пустой отчёт
Метрики читаются через `optional_metric`/`float()` (переваривает `"6.0"`), а `process/program/object_id` — через `int(row[6] or 0)`, который на строке `"6.0"` бросает `ValueError` → строка молча уходит в `skipped_rows` (стр. 616-618). Если панель когда-нибудь выгрузит `data_format_5..7` как REAL/текст с дробной частью — **отсеются ВСЕ строки, отчёт пустой**, причина скрыта в одном warning.

**Фикс:** `int(float(row[6] or 0))` для всех трёх кодов.

### 2. `webapp/app.py:3881-3885 → build_wash_detail (3217-3227)` — глобальный `state_lock` держится во время дискового I/O
Единственный обработчик, нарушающий паттерн остального кода: `build_wash_detail` целиком выполняется под `with state_lock`, а внутри — `load_app_settings()` (чтение+парсинг JSON с диска, стр. 3219) и оценка концентрации. Соседние `workspace_data` (3478-3489) и `wash_chart_data` (3888-3908) специально снимают снимок под локом и делают тяжёлую работу снаружи (с явным комментарием «на этом локе ждут SSE и все запросы»).

**Сценарий:** открытая карточка мойки + SSE-стрим `/api/workspace-job/stream` (берёт `state_lock` каждые 0.5 c) → чтение файла настроек под глобальным локом сериализует все запросы, подвисания UI на сетевом диске.

**Фикс:** снять снимок `analysis`/`revision` под локом, `build_wash_detail` вызывать вне лока — как в `wash_chart_data`.

### 3. `webapp/static/app.js:1293` — сортировка по имени объекта падает на `undefined`
`left.object.localeCompare(right.object, "ru")` — если сервер прислал строку без `object`/`null`, `TypeError` внутри `Array.sort` роняет `getFilteredRows` → список перестаёт перерисовываться (проявляется только в режиме сортировки `object_asc`; рендер строк безопасен через `escapeHtml`).

**Фикс:** `String(left.object || "").localeCompare(String(right.object || ""), "ru")`.

### 4. `webapp/static/app.js` — ни один `fetch` не имеет таймаута (нет `AbortController`)
Затрагивает все сетевые вызовы: `fetchWorkspaceData` (984), `getDetail` (1621), `getChartPayload` (1661), `fetchWorkspaceJobStatus` (271), `startWorkspaceRefresh` (1038), настройки (2138, 2147, 2160, 2172). Если бэкенд/FTP «завис» (TCP без RST) — промис не резолвится никогда: кнопка обновления вечно `disabled` с «Обновляю…» (3615-3618, `finally` не срабатывает), PDF залипает в «Сохраняю PDF…» (1989-1991), без индикации.

**Фикс:** хелпер-обёртка `fetch` с `AbortController` + `setTimeout` (15-30 c); по таймауту отклонять с понятной ошибкой и `showToast`/`setScreenError`.

## 🟡 Приоритет 2 — надёжность / edge-cases

### 5. `run_wash_desktop.py:1183-1234` — на холодном старте `.exe` окна нет несколько секунд
Окно/сплэш создаётся только ПОСЛЕ `load_web_app()` + `preflight_windows_runtime()` + `server.start()` (`wait_until_ready` timeout = 60 c). Красивый loading-HTML (193-312) фактически покрывает лишь микропаузу между `shown` и `load_url`. Приложение выглядит зависшим; если сервер «живой, но не отвечает» — до 60 c без окна.

**Фикс:** создавать и показывать loading-окно ПЕРВЫМ, старт сервера — в фоне.

### 6. `run_wash_ui.py:68-85` — ложный «Порт занят» в web-режиме
Берётся только `socket.getaddrinfo(host, port, ...)[0]`; при `HOST=localhost` первым по RFC 6724 может прийти `::1`, и при отключённом IPv6 — `probe.bind` → `OSError` → `SystemExit(1)`, хотя uvicorn поднялся бы на `127.0.0.1`. Дополнительно: нет `SO_REUSEADDR` (ложная занятость в TIME_WAIT после перезапуска); `PORT=0` делает проверку бессмысленной (проба и uvicorn возьмут разные случайные порты).

**Фикс:** перебирать все адреса из `getaddrinfo`; ставить `SO_REUSEADDR` на пробе; обрабатывать `PORT=0` явно.

### 7. `wash_report.py:697-718` — «сырой» минимум концентрации сегмента искажён краями фазы
На возврате концентрация нарастает с ~0 и спадает при вытеснении, поэтому `StatsBundle.minimum` сегмента щёлочи ≈ 0 даже при идеальной мойке. Сам вердикт `evaluate_concentration` (965-989) корректен — он обрезает края по первому/последнему превышению порога. Но если UI показывает raw-min как индикатор качества — ложная тревога на каждой норме.

**Фикс:** не показывать raw-min концентрации как метрику качества либо считать по «полке», как в `_evaluate_phase_concentration`.

### 8. `webapp/static/wash-chart.js:423, 634` — нет guard на `payload.panels`
`describePanels` сразу делает `payload.panels.map(...)`, а `mount` (634) проверяет только `has_data` и `Array.isArray(payload.series)`. **Смягчено:** текущий бэкенд (`chart_payload.py:120,162`) всегда кладёт `panels` в обе ветки — это защита от будущего рассинхрона, не активный краш.

**Фикс:** `(payload.panels || []).map(...)` либо добавить `panels` в guard.

### 9. `webapp/app.py:1890-1896` — shallow merge настроек затирает вложенные словари
`merged = {**load_app_settings(), **source}` — поверхностное слияние. POST с неполным `concentration_norms` (без второй фазы) или `result_labels` молча сбрасывает недостающие ключи в дефолт/`None` через `normalize_app_settings`. На практике фронт шлёт полный объект, но docstring обещает «частичное обновление».

**Фикс:** рекурсивный merge для `result_labels`/`concentration_norms` либо задокументировать «только целиком».

## Что проверено и НЕ является дефектом

- **Path traversal** (архивы + FTP) — закрыт двойной защитой.
- **CSRF / DNS rebinding / удалённый доступ** — middleware доверяет TCP-пиру, Origin проверяется на не-GET. Корректный выбор для local-only приложения.
- **IPv4-mapped loopback** (`::ffff:127.0.0.1`) — распознаётся корректно.
- **XSS** — все данные через `escapeHtml`; цвета через `isValidHexColorLike`; `app_state|tojson` безопасен.
- **Утечки памяти (фронт)** — таймеры/listeners корректно снимаются; кэши ограничены (`DETAIL_CACHE_LIMIT=200`, `CHART_PAYLOAD_CACHE_LIMIT=80`).
- **Гонки данных (фронт)** — guard'ы `workspaceHydrationToken`, `workspaceDataGeneration`, `modalRequestId` и др. отсекают устаревшие ответы.
- **Деления на ноль в графиках** — защищены `Math.max(…, 1)`/`1e-9`; `padRange`/`buildAxisTicks` обрабатывают `min===max`.
- **Hover** — троттлится через `requestAnimationFrame`.
- **Глотание исключений (бэк)** — стоят на осознанных best-effort операциях (закрытие FTP, chmod, unlink temp) с комментариями; сбои FTP/битые БД прокидываются в `scan_summary`/`warnings`.
- **Ресурсы (бэк)** — FTP в `finally`, файлы через `with`, `ThreadPoolExecutor` как контекст-менеджер.
- **`chart_payload.py:_downsample_points`** — корректен: монотонный x, верхняя граница точек соблюдена, экстремумы сохранены.
- **`read_samples`** — ловит повреждение БД и на `connect`, и на `fetch`.
